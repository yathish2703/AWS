Key concepts

  • Region: A physical location with multiple Availability Zones (e.g., us-east-1, eu-west-1)
  • Availability Zone (AZ): One or more isolated data centers with redundant power, networking, and connectivity
  • Edge Location: Data center used by CloudFront to cache content closer to users

Global vs regional services

Global services

Services that operate across all AWS regions automatically

IAM (Identity & Access Management)

Users, groups, roles work across all regions

Route 53

DNS service with global reach

CloudFront

CDN with 600+ edge locations worldwide

Organizations

Manage multiple AWS accounts globally

WAF (Web Application Firewall)

Global protection for web applications

Regional services

Services that must be deployed in specific AWS regions

EC2 (Elastic Compute Cloud)

Virtual servers run in specific regions/AZs

S3 (Simple Storage Service)

Buckets are created in specific regions (globally unique names)

RDS (Relational Database Service)

Databases run in specific regions/AZs

Lambda

Functions deployed per region

VPC (Virtual Private Cloud)

Networks are region-specific

Quick reference table

Service Type Scope Use case
IAM Global All regions User & permission management
EC2 Regional Per region/AZ Virtual servers
S3 Regional Per region (global namespace) Object storage
Route 53 Global All regions DNS service
CloudFront Global Edge locations worldwide Content delivery (CDN)
RDS Regional Per region/AZ Relational databases
Lambda Regional Per region Serverless functions
VPC Regional Per region Virtual networks
DynamoDB Regional Per region (global tables available) NoSQL database

Best practices

  • Choose regions close to your users to minimize latency
  • Use multiple Availability Zones for high availability and fault tolerance
  • Leverage global services like CloudFront for content delivery worldwide
  • Consider data residency requirements when selecting regions (GDPR, compliance)
  • Replicate critical data across regions for disaster recovery
