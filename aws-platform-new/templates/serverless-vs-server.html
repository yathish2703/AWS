Advantages & disadvantages

Server-based Advantages

  • ✓ Full control: Complete access to OS, libraries, and configurations
  • ✓ No time limits: Long-running processes supported
  • ✓ Predictable costs: Fixed monthly pricing for consistent workloads
  • ✓ Better for stateful apps: Keep data in memory between requests
  • ✓ Any runtime: Install any language, tools, or dependencies

Server-based Disadvantages

  • ✗ Always paying: Pay for idle time even with no traffic
  • ✗ Manual scaling: Need to configure Auto Scaling groups
  • ✗ Server maintenance: OS patches, security updates, monitoring
  • ✗ Capacity planning: Need to predict and provision capacity
  • ✗ Higher complexity: Manage infrastructure, networking, security

Serverless Advantages

  • ✓ Pay per use: Only pay for actual execution time (milliseconds)
  • ✓ Auto-scaling: Automatically scales from 0 to thousands
  • ✓ No maintenance: AWS manages servers, OS, patches
  • ✓ Built-in HA: High availability and fault tolerance included
  • ✓ Fast deployment: Deploy code in seconds, no infrastructure setup

Serverless Disadvantages

  • ✗ Cold starts: First request takes longer (100-1000ms delay)
  • ✗ Time limits: Maximum 15 minutes execution (Lambda)
  • ✗ Vendor lock-in: Tight coupling with AWS services
  • ✗ Limited resources: Max 10GB RAM, 512MB /tmp storage
  • ✗ Complex debugging: Harder to test and debug locally

Maintenance responsibilities

Server-based Maintenance

Your Responsibilities:

  • • OS patches and security updates
  • • Application deployment and updates
  • • Monitoring and logging setup
  • • Security group configuration
  • • Auto Scaling configuration
  • • Load balancer setup
  • • Backup and disaster recovery
  • • Performance optimization

AWS Responsibilities:

  • • Physical hardware maintenance
  • • Network infrastructure
  • • Hypervisor management
  • • Data center security

Serverless Maintenance

Your Responsibilities:

  • • Write and deploy function code
  • • Configure IAM permissions
  • • Set memory and timeout limits
  • • Monitor with CloudWatch Logs
  • • Manage dependencies in code

AWS Responsibilities:

  • • Physical hardware maintenance
  • • OS patches and security updates
  • • Automatic scaling
  • • High availability setup
  • • Network infrastructure
  • • Runtime environment
  • • Load balancing

When to use each approach

Use Server-based When:

  • • Long-running processes (> 15 minutes)
  • • Consistent high traffic (predictable load)
  • • Need full OS control or custom software
  • • Stateful applications with in-memory data
  • • Legacy applications that can't be refactored
  • • WebSocket connections or real-time apps
  • • High-performance computing workloads

Use Serverless When:

  • • Event-driven workloads (API, file uploads)
  • • Unpredictable or sporadic traffic
  • • Short-lived operations (< 15 minutes)
  • • Microservices and small independent functions
  • • Need rapid development and deployment
  • • Want to minimize operational overhead
  • • Cost optimization for low/variable traffic

Hybrid approach: Best of both worlds

Many modern applications use both server-based and serverless components together:

Core Application (EC2)

Main API and business logic running on EC2 instances

Background Tasks (Lambda)

Image processing, email sending, scheduled jobs

Event Processing (Lambda)

S3 uploads, database triggers, IoT data streams
