What is IAM?

AWS Identity and Access Management (IAM) is a web service that helps you securely control access to AWS resources. You use IAM to control who is authenticated (signed in) and authorized (has permissions) to use resources.

IAM Users

Individual people or applications that need AWS access

  • • Each user has unique credentials (username/password or access keys)
  • • Can be added to groups for easier management
  • • Best practice: One IAM user per person

IAM Groups

Collection of IAM users managed as a unit

  • • Example: Developers, Admins, Testers
  • • Users inherit permissions from their groups
  • • Easier to manage permissions for multiple users

IAM Policies

JSON documents that define permissions

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-bucket/*"
  }]
}
  • • Effect: Allow or Deny
  • • Action: What can be done (e.g., s3:PutObject)
  • • Resource: Which AWS resources

IAM Roles

Temporary credentials for AWS services or applications

  • • For AWS services (EC2, Lambda) to access other services
  • • More secure than hardcoding credentials
  • • Example: EC2 instance accessing S3

Authentication vs Authorization

Understanding the difference between authentication and authorization is fundamental to AWS security.

Authentication

AuthN

"Who are you?"

Verifying identity

Authentication Methods:

  • • Username & Password - Console access
  • • Access Keys - Programmatic access (API/CLI)
  • • MFA (Multi-Factor) - Extra security layer
  • • Temporary Tokens - From assumed roles
  • • Federated Identity - Corporate SSO

Example:

🔐 Login to AWS Console

→ Enter username: alice

→ Enter password: ••••••••

✅ Identity verified!

Authorization

AuthZ

"What can you do?"

Checking permissions

Defined by IAM Policies:

  • • Effect - Allow or Deny actions
  • • Action - What operations (s3:GetObject)
  • • Resource - Which AWS resources (ARN)
  • • Condition - When it applies (optional)

Example:

📋 Alice tries to list S3 objects

→ AWS checks policies...

→ Policy allows s3:ListBucket

✅ Action authorized!

Key differences

Aspect Authentication Authorization
Question Who are you? What can you do?
Purpose Verify identity Grant permissions
When First (at login) After authentication
AWS Examples Password, access keys, MFA IAM policies, resource policies
Error Message "Invalid credentials" "Access denied"

Common use cases

1

Control team member access

Create IAM users for each team member with specific permissions based on their role

Example:

  • • Developers: Read/write access to EC2, S3
  • • Admins: Full access to all services
  • • Auditors: Read-only access for monitoring
2

Secure application access

Use IAM roles for EC2 instances to access other AWS services without storing credentials

Scenario:

Your web app on EC2 needs to upload files to S3. Instead of hardcoding AWS keys, attach an IAM role to the EC2 instance with S3 write permissions.

3

Cross-account access

Allow users from one AWS account to access resources in another account

Example: Production account users need read-only access to development account for troubleshooting. Create a cross-account IAM role.

IAM security best practices

Enable MFA (Multi-Factor Authentication)

Add extra layer of security for root and IAM users

Principle of least privilege

Grant only the minimum permissions needed

Use groups to assign permissions

Easier to manage than individual user policies

Rotate credentials regularly

Change passwords and access keys periodically

Don't use root account

Create IAM users instead, lock root account with MFA

Monitor with CloudTrail

Track all IAM actions for security auditing
